# 🚀 Custom Robotic Arm: ROS 2 Jazzy & MoveIt 2 Troubleshooting Guide

This document archives all the critical issues faced, debugged, and solved while porting a custom 5-DOF (+ gripper) robotic arm from Fusion 360 into a fully operational ROS 2 Jazzy, Gazebo Harmonic, and MoveIt 2 environment. 

---

## 🛑 Issue #1: Gazebo Meshes Failing to Load
**Observed Behavior:** When launching Gazebo, the robot appears completely invisible or throws `URI not found` errors.
**In-Depth Explanation:** ROS 2 uses the `ament` build system. When an STL or DAE mesh is nested in the `src/` folder, the environment doesn't natively know it's there. Gazebo specifically looks into environment variables to find graphical assets. If the path from your `<mesh filename="package://...">` cannot be evaluated, Gazebo drops the visual.
**Manual Solution:** 
You must point `GZ_SIM_RESOURCE_PATH` to the `install/share` directory generated by `colcon build`.
```sh
# Run this before launching your gazebo simulation
export GZ_SIM_RESOURCE_PATH=$GZ_SIM_RESOURCE_PATH:/home/natraj/file/install/robot_arm_description/share
```
**Testing Rule:** Re-launch Gazebo. The physical mesh of the robot should instantly appear.

## 🛑 Issue #2: The Simulation "Explodes" or Drops the Gripper Fingers
**Observed Behavior:** The robot spawns, but the moment physics initializes, the gripper flies off, or the simulator crashes entirely.
**In-Depth Explanation:** Your URDF features a "parallel jaw gripper" mimicking a main actuation joint using `<mimic>` tags. As of ROS 2 Jazzy, the default Gazebo Harmonic physics engine (DART) struggles heavily with closed-loop kinematic chains and mimic joints. 
**Manual Solution:** 
We had to switch the underlying physics engine loaded by Gazebo to `bullet-featherstone` which specifically supports complex joint constraints natively without tearing the model apart.
Update your launch file arguments to inject the physics plugin:
```python
# In gazebo.launch.py
gz_args = ['-v 4 -r empty.sdf --physics-engine gz-physics-bullet-featherstone-plugin']
```
**Testing Rule:** Spawn the URDF. The parallel fingers should stay securely attached and simulate gravity properly.

## 🛑 Issue #3: Incompatible URDF Tags for ROS 2 Control
**Observed Behavior:** MoveIt and Gazebo complained about unidentified hardware interfaces, or the generated URDF refused to load controllers.
**In-Depth Explanation:** Fusion 360 URDF exporters generate old ROS 1 `<transmission>` tags. ROS 2 completely retired this format in favor of `<ros2_control>` tags. Furthermore, an active joint requires both `command_interface` AND `state_interface`, but a Mimic joint *only* requires a `state_interface` (because it cannot be actively commanded by a motor, it physically follows another joint).
**Manual Solution:** 
We deleted the legacy `<transmission>` tags and injected:
```xml
<ros2_control name="GazeboSystem" type="system">
  <!-- Active Joint -->
  <joint name="link_1">
    <command_interface name="position"/>
    <state_interface name="position"/>
  </joint>
  <!-- Passive Mimic Joint -->
  <joint name="link_10">
    <state_interface name="position"/>
  </joint>
</ros2_control>
```
**Testing Rule:** Check the `/joint_states` ROS topic to see all joints successfully echoing position data.

## 🛑 Issue #4: Xacro Crashes on Empty YAML (`NoneType` Error)
**Observed Behavior:** Running MoveIt's Setup Assistant launch files caused an immediate Python `TypeError: unsupported operand type(s) for *: 'NoneType' and 'float'`.
**In-Depth Explanation:** The MoveIt Setup Assistant automatically generated an `initial_positions.yaml` file but left the values empty (null). When the `ros2_control` xacro parser tried to inject these default values into the URDF, it choked on the empty dictionary.
**Manual Solution:** 
Open `config/initial_positions.yaml` and manually add `0.0` or standard starting angles for every joint.
```yaml
initial_positions:
  link_1: 0.0
  link_2: 0.0
  # (continue for all joints)
```

## 🛑 Issue #5: MoveIt RViz "Unable to Plan" (Zero Velocity Limits)
**Observed Behavior:** Generating a path in RViz fails silently. The robot won't move when "Plan" is pressed.
**In-Depth Explanation:** MoveIt's trajectory generation algorithms (like Time-Optimal Trajectory Parameterization) require maximum velocities and accelerations to calculate a physical path over time. The generated `joint_limits.yaml` overrode the URDF limits, forcing `max_velocity: 0` and `max_acceleration: 0` for all joints. Because the permitted speed was zero, the math solver instantly failed.
**Manual Solution:** 
Update `config/joint_limits.yaml` to enable the boundaries:
```yaml
joint_limits:
  link_1:
    has_velocity_limits: true
    max_velocity: 1.0
    has_acceleration_limits: true
    max_acceleration: 1.0
```
**Testing Rule:** Drag the marker in RViz and click "Plan". The orange ghost arm preview should correctly animate the planned path.

## 🛑 Issue #6: "No action namespace specified" for Controllers
**Observed Behavior:** Even if a plan generates, hitting "Execute" fails, and terminal errors say: `No action namespace specified for controller 'arm_controller'`.
**In-Depth Explanation:** MoveIt needs to know exactly which ROS Action namespace the fake (or real) controllers are listening to so it can dispatch the trajectory point cloud. 
**Manual Solution:** 
Edit `config/moveit_controllers.yaml` to explicitly define the `action_ns` for your FollowJointTrajectory controllers.
```yaml
  arm_controller:
    action_ns: follow_joint_trajectory
    default: true
    type: FollowJointTrajectory
```

## 🛑 Issue #7: The Robot Phases Through the Ground
**Observed Behavior:** Dragging an interactive marker allows the arm to plan a path right through the grid floor.
**In-Depth Explanation:** By default, the MoveIt Collision Matrix only cares about the robot's links hitting *each other*. The simulation environment is considered empty space unless instructed otherwise. 
**Manual Solution (in RViz):** 
1. Go to **MotionPlanning** panel → **Scene Objects** tab.
2. Click **+** (Add) to drop a Box in the scene.
3. Scale the XYZ to `2.0, 2.0, 0.1` and set its Z-position slightly below zero (e.g., `-0.06`).
4. Click **Publish Scene**.
**Testing Rule:** Try dragging the robot ball under the floor. The exact collision point should light up red, and planning will fail, successfully saving the virtual (and real) arm from floor damage.

## 🛑 Issue #8: The RGB Marker (Ball) Gets Stuck / Rotates Improperly
**Observed Behavior:** When trying to move the robot base backwards, the ball refuses to move, snaps back, or feels highly restricted.
**In-Depth Explanation:** Your robot is a **5-DOF** arm. MoveIt's default Inverse Kinematics (IK) solver is **KDL**, which expects a full **6-DOF** coordinate (XYZ + Roll/Pitch/Yaw) from the interactive marker. Because a 5-DOF arm physically cannot satisfy full 6-DOF requests at arbitrary points, the math solver hits a "singularity", assumes the position is physically impossible, and refuses to render the move.
**Manual Solution:** 
We must instruct the KDL solver to prioritize hitting the *XYZ coordinates* while ignoring the unachievable *wrist orientation*.
In `config/kinematics.yaml`, add:
```yaml
arm:
  kinematics_solver: kdl_kinematics_plugin/KDLKinematicsPlugin
  position_only_ik: True
```
**Testing Rule:** Reload MoveIt. Dragging the center of the ball will now feel buttery smooth because the solver generates IK paths based only on the XYZ translation of the target, safely ignoring the marker's rotational arrow restraints.
