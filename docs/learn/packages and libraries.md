Here is a comprehensive breakdown of the core packages and libraries utilized throughout your robotic arm project, categorized by their engineering domain. This will give you clarity on the architectural choices and functionality of your tech stack.

### 1. ROS 2 Core & Middleware Modules
These are the foundational systems that allow your separated scripts to talk to each other.
*   **`rclpy` / `rclcpp`:** The standard ROS 2 client libraries for Python and C++. 
    *   *Functionality:* Provides the API to create Nodes, Publishers, Subscribers, and Timers. It handles the underlying DDS (Data Distribution Service) network communication.
*   **`std_msgs`, `geometry_msgs`, `sensor_msgs`, `trajectory_msgs`:** ROS 2 standard message interfaces.
    *   *Functionality:* Defines the strict data structures exchanged across topics. For example, `sensor_msgs/Image` for raw camera frames, `geometry_msgs/PoseStamped` for 3D coordinates, and `trajectory_msgs/JointTrajectory` to tell the arm servos exactly where to move and how fast.

### 2. Simulation & Modeling (Gazebo Harmonic)
These packages bridge the mathematical model of your robot into a physics-based visual simulation.
*   **`xacro` (XML Macros):**
    *   *Functionality:* A macro language used to organize your URDF (`robot.urdf.xacro`). It allows you to use variables and math inside the XML file (e.g., automatically calculating inertia matrices instead of hardcoding them) to keep the robot description clean.
*   **`ros_gz_sim` & `ros_gz_bridge`:**
    *   *Functionality:* The critical bridge between ROS 2 Jazzy and Gazebo Harmonic. Gazebo uses its own internal messaging system (Ignition Transport); the bridge translates ROS 2 topics (like `/joint_states`) into Gazebo topics and passes the simulation clock back to ROS 2.
*   **`gz-sim-joint-state-publisher-system`:**
    *   *Functionality:* A Gazebo C++ system plugin attached to your URDF. It actively reads the physics engine's joint angles and publishes them so RViz and MoveIt know the exact real-time posture of the robot in the simulation.

### 3. Motion Planning & Control (MoveIt 2)
These libraries perform the complex mathematics required to move a 6-DOF arm without it crashing into itself or the environment.
*   **`moveit_core` & `moveit_ros_planning_interface`:**
    *   *Functionality:* The C++/Python brain of MoveIt 2. It contains the KDL (Kinematics and Dynamics Library) solver which calculates Inverse Kinematics. You use the `MoveGroup` API here to send a target 3D coordinate and let it return a safe array of joint angles.
*   **`ros2_control` & `controller_manager`:**
    *   *Functionality:* The framework that abstracts hardware away from the planner. MoveIt asks for a path, and `ros2_control` ensures those paths are sent to either the Gazebo simulation or your physical Arduino via generic "interfaces."
*   **`joint_trajectory_controller`:**
    *   *Functionality:* A specific plugin inside `ros2_control` that takes the joint path generated by MoveIt and converts it into smooth, time-parameterized splines using PID algorithms, ensuring the physical servos don't violently jerk.

### 4. Computer Vision & AI (Autonomous Mode)
The "Vision-Guided" portion of your thesis relies on these edge-AI packages.
*   **`opencv-python` (`cv2`):**
    *   *Functionality:* Open Source Computer Vision Library. Used in `vision_node.py` to interface with the camera, read raw pixel matrices, convert color spaces (BGR to HSV), and draw bounding boxes.
*   **`ultralytics` (YOLOv8):**
    *   *Functionality:* The state-of-the-art machine learning package for object detection. It takes the OpenCV image frame, runs a neural network inference, and outputs exactly where objects (like colored blocks) are located in 2D space.
*   **`torch` (PyTorch):**
    *   *Functionality:* The underlying tensor/matrix compute engine that powers YOLOv8. It handles processing the neural network on the CPU or pushing it to a GPU (CUDA) for high-speed inference.

### 5. Human-in-the-Loop & Networking (Teleoperation)
These packages facilitate the two local and remote override modes.
*   **`mediapipe` (by Google):**
    *   *Functionality:* A cross-platform ML framework used in `gesture_node.py`. The `HandLandmarker` model specifically extracts 21 precise 3D (x, y, z) spatial coordinates from a human hand in real-time. You mathematically map these vectors to your robot's joint angles.
*   **`paho-mqtt`:**
    *   *Functionality:* An Eclipse Foundation library used in `mqtt_bridge_node.py`. It implements the lightweight MQTT IoT protocol. It allows your robot to act as a client that listens to a public broker (like HiveMQ Cloud) to receive teleoperation commands from a web app anywhere in the world.

### 6. Embedded Hardware (Physical Actuation)
The lowest-level system that physically turns electrical signals into torque.
*   **`pluginlib` & POSIX C++ headers (`<termios.h>`):**
    *   *Functionality:* Inside your `robot_arm_hardware/system_interface.cpp`, `pluginlib` dynamically loads your custom C++ script into standard ROS 2. `<termios.h>` is the Linux system library used to open the native USB port and securely stream serial bytes (`115200` baud) to the Arduino.
*   **Adafruit PWM Servo Driver Library (Arduino IDE):**
    *   *Functionality:* Used inside `firmware.ino` on the microcontroller. It communicates over I2C (SDA/SCL pins) to the PCA9685 16-channel driver board, turning the ROS 2 degree/radian values into exact microseconds of Pulse Width Modulation (PWM) to actuate the DC servo motors.
